## 精简错误信息
- 提交 `4376cc3` 在 Analyze / Test 三个任务里加了 `flutter config --enable-swift-package-manager`，但 CI 仍在 `flutter pub get` 失败。
- 真实根因不是「未启用 SPM」，而是 Linux runner 无 Xcode：`usesSwiftPackageManager` 在无 Xcode 15+ 时恒为 false，启用配置无效。
- `receive_sharing_intent: ^1.8.0` 在 CI（默认 pub.dev）可解析到 **1.9.0**；1.9.0 移除了 CocoaPods podspec，仅 SPM，触发：
  `Plugin receive_sharing_intent is only Swift Package Manager compatible...`
- 本地 `pubspec.lock` 锁的是 1.8.1 且 hosted 于 `pub.flutter-io.cn`；CI 按 pub.dev 校验/解析时会升到 1.9.0。
- 失败 run：`https://github.com/sprogFall/life_tools/actions/runs/30015337408`

## 解决方案
- 将 `receive_sharing_intent` **精确固定为 `1.8.1`**（仍含 podspec，Linux CI 可过）。
- 回滚 workflow 中无效的 `flutter config --enable-swift-package-manager` 步骤。
- 更新 `scripts/tests/build-apk_workflow_test.sh`：断言 workflow **不包含** 该 SPM 配置命令。
- 验证命令：
  - `bash scripts/tests/build-apk_workflow_test.sh`
  - `PUB_HOSTED_URL=https://pub.dev flutter pub get`（应解析到 1.8.1 且无 SPM 报错）
  - `bash scripts/pre-push.sh --scope flutter --test-module core`（或按改动范围）

## 预防方案
- 对存在「新版本破坏 CI 平台兼容性」的插件，避免宽松 `^` 上限，优先精确版本或上限约束（如 `<1.9.0`）。
- 新增/升级 iOS 插件后，必须在 **Linux + pub.dev** 环境跑一次 `flutter pub get`，不能只在国内镜像/本机锁文件下验证。
- 不要在 Linux Android CI 上依赖 `flutter config --enable-swift-package-manager` 解决 SPM-only 插件问题：无 Xcode 时该开关无法生效。
- 升级到 `receive_sharing_intent` 1.9+ 前，需确认 Flutter 工具在无 Xcode 主机上已跳过 Darwin 插件校验，并评估 iOS SceneDelegate/SPM 迁移成本。
